# Пояснение: Задача 1.2 — Алгоритмы и решения

## Почему такая структура DAG?

DAG строит цепочку: `turnover[01.01] → balance[01.01] → turnover[02.01] → ...`

**Причина строгой последовательности:**  
Остаток за день N зависит от остатка за день N-1. Если запустить дни параллельно — balance[02.01] может посчитаться раньше чем balance[01.01], и возьмёт нулевой остаток за предыдущий день.

## Почему FULL JOIN для оборотов?

Один счёт может быть:
- Только кредитовым (поступление)
- Только дебетовым (списание)
- Обоими в один день

`FULL JOIN` гарантирует что счёт попадёт в витрину в любом случае.  
`LEFT JOIN` потерял бы счета, которые участвовали только как дебетовые.

## Активные и пассивные счета — почему разная формула?

Это банковская логика двойной записи:

- **Активный счёт** (имущество банка): растёт по дебету, уменьшается по кредиту
  - Пример: кредит выданный клиенту — дебет увеличивает задолженность
- **Пассивный счёт** (обязательства банка): растёт по кредиту, уменьшается по дебету
  - Пример: вклад клиента — кредит увеличивает обязательство перед клиентом

## Почему стартовые остатки за 31.12.2017?

Расчёт остатков — цепная формула. Каждый день опирается на предыдущий.  
Для 01.01.2018 нужен остаток за 31.12.2017 — он берётся из DS.FT_BALANCE_F (исходные данные).

## Логирование — выбор подхода

Используем ту же таблицу `logs.etl_log`, что и в задаче 1.1, с полями:
- `start_time` / `end_time` — время старта и окончания расчёта
- `table_name` — какая витрина рассчитывается
- `status` — 'started' / 'success' / 'error'
- `rows_loaded` — сколько записей вставлено
- `error_message` — текст ошибки если что-то пошло не так

**Почему именно так:**
1. Единая таблица логов для всего проекта — удобно мониторить
2. Паттерн "запись при старте + обновление при финише" — видно зависшие расчёты (статус 'started' без end_time)
3. Идемпотентность: удаление данных за дату + лог позволяют отследить перезапуски

## Проверка результата

После запуска DAG в каждой витрине должны появиться записи за все 31 день января 2018.  
В `logs.etl_log` должно быть 62 записи (31 день × 2 витрины) со статусом 'success'.
